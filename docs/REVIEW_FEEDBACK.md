# Review Feedback

PALISADE is still in expert-preview mode. The most useful feedback is specific, concrete, and tied to real operator needs or detection quality.

## What Feedback Is Most Useful

Please focus on:

- whether the problem framing is real
- whether the current differentiation is credible
- whether the outputs would be useful to an operator
- where the tool overclaims or creates false confidence
- where adoption would likely fail in practice
- where a finding looks misleading, weak, or incomplete

## What To Comment On

Useful areas for feedback:

- product positioning
- operator workflow
- detection quality
- report usefulness
- control/framework alignment
- trust and adoption barriers

## What To Include

If you file feedback, please include:

- your role or reviewer type
- the environment or use case you are thinking about
- what you tested or reviewed
- what felt useful
- what felt unclear or weak
- what you would not trust yet
- the single highest-value improvement

## Detection-Oriented Feedback

If your feedback is about a specific vendor or finding, include:

- vendor and product
- observed behavior
- expected behavior
- whether this looked like a false positive, false negative, or confidence problem
- any artifact, fixture, screenshot, or report excerpt that supports the feedback

## Current Review Lens

At this stage, the highest-value questions are:

1. Is PALISADE solving a real enough problem?
2. Is it clearly different from existing public services and generic scanners?
3. Are the findings honest and useful enough to support operator action?
4. What would block adoption even if the detection is technically sound?

## Not Helpful

Less helpful feedback includes:

- generic encouragement without specifics
- requests to broaden PALISADE into a full OT platform
- feature ideas without a clear operator use case
- criticism that does not distinguish product scope from implementation bugs
